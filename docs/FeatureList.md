# 本地电子阅读器功能清单

本文档基于 `docs/Feasibility.md` 的技术可行性分析，整理一份面向 Electron + React + TypeScript + shadcn/ui 技术栈的本地电子阅读器功能清单。目标是先构建稳定可用的离线阅读器，再逐步扩展多格式、全文搜索、OCR、TTS、同步等高级能力。

## 一、产品定位

### 1.1 核心定位

- 一款跨平台、纯本地优先的桌面电子书阅读器。
- 支持主流电子书、文档、漫画与扫描书格式。
- 阅读数据默认只保存在本机，不依赖账号、不强制联网、不上传书籍内容。
- 以现代化 UI、良好的阅读体验和广泛格式兼容作为主要差异化。

### 1.2 目标用户

- 有大量本地电子书、PDF、漫画或资料文档的用户。
- 重视隐私、离线使用和本地数据掌控的用户。
- 需要统一管理 EPUB、PDF、TXT、MOBI、AZW3、CBZ 等多种格式的用户。
- 希望获得比传统阅读器更现代界面和更顺滑体验的桌面端用户。

### 1.3 产品原则

- 本地优先：所有核心功能离线可用。
- 隐私优先：书籍文件、阅读记录、笔记、搜索索引默认不出本机。
- 渐进增强：在线翻译、在线 TTS、WebDAV 同步等功能必须是可选项。
- 格式优先：先保证 EPUB、PDF、TXT 的稳定体验，再逐步扩展长尾格式。
- 性能优先：大文件 PDF、超长 TXT、大书库搜索必须有明确优化策略。
- 可扩展优先：格式解析、阅读器渲染、元数据提取、同步能力应模块化。

## 二、功能优先级说明

| 优先级 | 含义 |
|---|---|
| P0 | MVP 必备功能，没有它产品无法作为阅读器使用 |
| P1 | 核心体验功能，决定是否好用 |
| P2 | 高级功能，形成差异化竞争力 |
| P3 | 长期增强功能，适合后续版本迭代 |

## 三、书库管理

- [ ] P2 支持监听本地书库文件夹变化并自动更新。

- [ ] P3 支持 OPDS 书源浏览与下载，但默认不作为核心书源能力。

- [ ] P2 支持紧凑模式与大封面模式。

## 四、格式支持

### 4.1 MVP 格式

- [ ] P0 EPUB 阅读支持，优先基于 `foliate-js`。
- [ ] P0 TXT 阅读支持，支持常见编码识别与解码。
- [ ] P0 PDF 阅读支持，基于 `pdf.js`。

### 4.2 核心扩展格式

- [ ] P1 MOBI 阅读支持，优先使用 `foliate-js`。
- [ ] P1 AZW3 阅读支持，优先使用 `foliate-js`。
- [ ] P1 FB2 阅读支持，基于 XML 解析。
- [ ] P1 CBZ 漫画阅读支持，基于 `jszip`。
- [ ] P2 CBR 漫画阅读支持，基于 `unrar` 相关方案。
- [ ] P2 DJVU 阅读支持，基于 `djvu.js`。
- [ ] P2 CHM 阅读支持，作为资料类格式补充。
- [ ] P3 UMD 等小众格式支持。

### 4.3 格式兼容与兜底

- [ ] P1 不支持的格式需要显示清晰错误提示。
- [ ] P1 解析失败时保留失败日志，方便排查。
- [ ] P2 MOBI/AZW3 解析失败时可选调用 Calibre CLI 转换为 EPUB。
- [ ] P2 检测系统是否安装 Calibre，并在需要时引导用户配置。
- [ ] P3 支持格式转换任务队列。
- [ ] P3 支持将部分格式导出为 EPUB、PDF 或 TXT。

## 二十三、暂不纳入首版的功能

- 在线书城和内容分发。
- 用户账号系统。
- 社交、评论、社区和排行榜。
- 云端强制同步。
- DRM 破解或受版权保护内容绕过。
- 复杂排版编辑器。
- 移动端首发。

## 二十四、关键风险提醒

- PDF 大文件性能是首要风险，需要从 MVP 阶段就使用 Worker 和按需渲染思路。
- EPUB 渲染不要从零实现完整排版引擎，应优先参考 `foliate-js` 的成熟方案。
- MOBI/AZW3 兼容性可能不稳定，需要准备 Calibre CLI 兜底。
- OCR 是亮点但实现成本较高，适合放到 P2 阶段。
- Electron 体积无法完全避免，应把目标定在合理范围内，而不是追求极限小体积。
- 纯本地产品的在线能力必须克制，任何联网功能都应明确、可关闭、可解释。
